# MenuRay — Stitch 提示词包

> **使用流程：**
> 1. 新 Stitch 会话先粘贴 [`DESIGN.md`](./DESIGN.md) 全文 — 让模型先理解品牌系统
> 2. 然后按下面 **11 个批次** 顺序丢，一次发一个批次
> 3. 不满意用末尾的 **"迭代提示模板"** 单屏微调
>
> **依据：** Stitch 官方 prompt guide（plain language, screen by screen, 单条 prompt 控制在 5000 字以内）+ 社区 March 2026 多屏经验（甜蜜区间 3–4 屏 / 批次）。
>
> **每个批次内嵌"Zoom-Out-Zoom-In"结构：** 先讲流的目的（zoom-out），再列每屏细节（zoom-in）。

## 批次 1 — A1 登录 / 注册

```
为 MenuRay 商家端生成"登录 / 注册"屏（移动端原生）。

这是商家进入 App 的第一屏，需传递温暖、简洁、可信任的第一印象。

屏幕内容：
- 顶部：Logo（墨绿底 + 暖米白菜单 + 翻起一角）+ 字标 "MenuRay" + slogan「拍一张照，5 分钟生成电子菜单」
- 表单：手机号输入框；验证码输入框（右侧含"获取验证码"按钮，60 秒倒计时）
- 主按钮："登录"
- 切换链接："新用户？立即注册"
- 底部：用户协议 + 隐私政策链接

状态：
- 验证码倒计时进行中：按钮灰色 + "59s 重发"
- 验证码错误：输入框下红色提示
- 登录成功：跳转下一屏

严格使用 DESIGN.md 中定义的品牌色与字体规范。
```

## 批次 3 — A3–A6 导入流（4 屏连贯）

```
为 MenuRay 商家端生成"导入菜单"流程的 4 个连续屏（移动端原生），按顺序依次连接，要求视觉与交互高度连贯。

流程目的：商家把纸质菜单变成结构化数字数据。从拍照开始，经过校正 + OCR，最终得到可编辑的菜品列表。

【屏 A3：拍摄菜单照片】
- 全屏相机取景框，实时菜单边缘检测（黄色虚线框跟随，检测成功时变绿框 + 提示"可以拍了"）
- 顶部：关闭按钮、闪光灯切换、"换相册上传"链接
- 中部气泡："对准菜单边缘，自动校正"
- 底部：已拍缩略图条 + 大快门按钮 + "完成 (3)" 按钮
- 光线不足态：顶部黄色提示条"光线偏暗，建议开启闪光灯"

【屏 A4：从相册上传菜单】（A3 的切换路径）
- 顶部：取消、标题"选择菜单图片"、右上"下一步 (3)"
- 相册图片网格，每行 4 张
- 已选图片右上角显示选择序号 ①②③
- 底部已选缩略图条可拖拽排序
- 状态：未选时"下一步"灰色，选中至少 1 张时高亮

【屏 A5：图像校正与裁剪】
- 顶部：返回、标题"校正图片 (1 / 3)"、右上"下一步"
- 中部大图 + 4 个可拖动角点 + 校正后预览叠层
- 底部工具栏：自动校正、旋转、裁剪、对比度增强、撤销
- 底部缩略图条，当前编辑张高亮
- 自动校正态：菱形 loading 动画 + "智能校正中"

【屏 A6：OCR 识别中】
- 居中插画：菜单图 → 文字流动 → 菜品卡片，循环动画
- 分阶段进度文案："正在识别文字 (1/3 张)" → "正在识别菜品结构" → "正在归类整理"
- 进度条
- 底部小字："识别用时约 30 秒，可点'后台运行'继续操作"
- 主按钮"后台运行" + 次按钮"取消"
- 失败态：进度条变红 + "识别失败，请重试" + 重试 / 返回按钮

严格使用 DESIGN.md 的品牌色与字体规范。
```

## 批次 4 — A7 识别结果编辑

```
为 MenuRay 商家端生成"识别结果编辑"屏（移动端原生）。

这是导入流后的关键屏 — OCR 自动识别完成后，商家在此核对、修正菜品。信息密度高，需要清晰的视觉层级。

屏幕内容：
- 顶部：返回、标题"整理菜单"、右上"下一步"

- 每行菜品：菜名 + 价格 + 小图标（标签提示 / 翻译已加 / 配图已加）
- 低置信度行：左侧橙色竖条 + 末尾 "?" 提醒（暗示"需要核对"）
- 右下浮动按钮："+ 新增"

示例数据（原样使用）：
- 类别 "凉菜"：
  - 口水鸡 ¥38
  - 凉拌黄瓜 ¥18
  - 川北凉粉 ¥22 ← 这一行是低置信度，左侧橙色竖条 + 行尾 "?"
- 类别 "热菜"：
  - 宫保鸡丁 ¥48
  - 麻婆豆腐 ¥32

交互态：
- 编辑某行：行变蓝边、底部弹键盘
- 拖拽排序：行半透明跟随手指
- 滑动删除：行右侧露出红色"删除"按钮
- 空类别：显示 "+ 添加菜品"

严格使用 DESIGN.md 的品牌色与字体规范。
```

## 使用建议（实操经验）

1. **第一批先丢 DESIGN.md**，等 Stitch 确认理解后再丢批次 1
2. **批次 2 是关键调试点**：A2 首页生成后看整体调子，不对就改 DESIGN.md 重来
3. **每批之间停一下看结果**，不要连发 — 出问题及时校正
4. **同一会话别超 ~6 个批次**，超过就开新会话（重新粘贴 DESIGN.md），避免上下文漂移
5. **示例数据保留**：里面的"云间小厨"、"宫保鸡丁 ¥48" 等都是有意写的，不要换 Lorem Ipsum
6. **状态变量分别生成**：复杂状态（loading / error / empty）建议用迭代提示单独再生成
7. **导出**：满意后用 Stitch 自带的 Figma 导出或代码导出
