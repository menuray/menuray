# Internationalization (i18n)

> How MenuRay handles multiple languages — both for the **UI chrome** (buttons, labels, headers) and for **dynamic menu content** (dish names, descriptions translated per dish).

These are two very different problems with two different solutions.

---

## 1. UI chrome (static strings)

The merchant app and customer view contain a fixed set of UI strings ("Save", "Add menu", "Sold out"). These get translated once and shipped with the app.

### Strategy

- **Default locale: English** (`en`).
- **First supplemental locale: Chinese Simplified** (`zh-CN`) — to retain existing Chinese copy from initial development.
- **Add more locales** as community translations come in.
- Use Flutter's official **`flutter_localizations` + `intl`** + **`.arb`** files.

### Status

🚧 **Not yet implemented.** Current screens have hardcoded Chinese strings. Migration is on the roadmap (see [roadmap.md](roadmap.md)) and is a precondition for the public OSS launch announcement.

### Once implemented: file layout

```
frontend/merchant/lib/l10n/
├── intl_en.arb       # English (template / source of truth)
├── intl_zh.arb       # Chinese Simplified
├── intl_ja.arb       # Japanese (community contribution)
├── intl_es.arb       # Spanish (community contribution)
└── ...
```

`pubspec.yaml`:
```yaml
flutter:
  generate: true
  
flutter_intl:
  enabled: true
  arb_dir: lib/l10n
  output_dir: lib/generated
```

### Key naming convention

Use `dot.notation` grouped by feature:
```json
{
  "common.save": "Save",
  "common.cancel": "Cancel",
  "auth.login.title": "Welcome",
  "auth.login.phoneHint": "Phone number",
  "home.menus.title": "My menus",
  "home.menus.empty": "No menus yet — snap one!",
  "home.menus.cardSubtitle": "{count} views",
  "@home.menus.cardSubtitle": {
    "placeholders": { "count": { "type": "int" } }
  }
}
```

Rules:
- Keys are stable English-style identifiers.
- Use `{placeholder}` for dynamic values — never concatenate strings in code.
- Plurals via `intl` plural syntax.

### Contributor flow for a new language

1. Copy `lib/l10n/intl_en.arb` → `lib/l10n/intl_<locale>.arb` (e.g., `intl_fr.arb` for French).
2. Translate every value. Keep keys + `{placeholders}` exactly as in source.
3. Add the locale to the `supportedLocales` list in `app.dart`.
4. Run `flutter pub get` to regenerate the bindings.
5. Test by changing system language or via the in-app language picker (when added).
6. Submit PR.

### How the app picks a locale

1. User's **explicit choice** (saved in app preferences) wins if set.
2. Otherwise, **OS locale** (via `WidgetsBinding.instance.platformDispatcher.locale`).
3. Falls back to **English** if no supported locale matches.

## 2. Dynamic menu content (per-dish translations)

Menu content is created by merchants and translated per dish. This is **dynamic data**, stored per-row in the database — not part of the static `.arb` bundle.

### Data shape (planned, in Postgres)

```sql
CREATE TABLE dish_translations (
  id            uuid PRIMARY KEY,
  dish_id       uuid NOT NULL REFERENCES dishes(id) ON DELETE CASCADE,
  locale        text NOT NULL,           -- e.g., 'en', 'zh-CN', 'ja'
  name          text NOT NULL,
  description   text,
  created_at    timestamptz DEFAULT now(),
  updated_at    timestamptz DEFAULT now(),
  UNIQUE (dish_id, locale)
);

CREATE INDEX dish_translations_dish_locale_idx ON dish_translations (dish_id, locale);
```

Same pattern for `category_translations`, `store_translations`.

### How translations are produced

1. **Source language**: whatever the merchant entered when creating the dish (commonly Chinese or English).
2. **Other languages**: the merchant clicks "Translate" → an Edge Function calls an LLM (Claude/GPT) → the result is stored in `dish_translations`.
3. **Manual override**: merchants can edit any translated row.

### How customer view picks a translation

1. User's URL param `?lang=ja` wins if present.
2. Else: browser `Accept-Language` header negotiated against available `dish_translations.locale` for the menu.
3. Falls back to the menu's **source locale** if no translation exists.

### Dish-level languages don't have to mirror UI-chrome languages

Customer view UI chrome may be in (say) Japanese, while a dish that hasn't been translated falls back to its source (Chinese). UI clearly indicates the fallback ("Original: 中文") so the diner can ask staff.

## Currency, dates, numbers

- **Currency**: stored as `(amount: numeric, currency: text)` per menu — `currency` is an ISO 4217 code (`CNY`, `USD`, `EUR`, `JPY`). Display formatting via `intl.NumberFormat.simpleCurrency(locale: ..., name: currency)`.
- **Dates / times**: store UTC, render in user-local time (Flutter `intl.DateFormat` handles this).
- **Numbers** (view counts, etc.): `intl.NumberFormat.decimalPattern(locale)` — handles thousand separators per locale.

## Right-to-left languages (RTL)

Flutter handles RTL layout automatically when the locale is RTL (e.g., Arabic, Hebrew). Things to remember:
- Use `EdgeInsetsDirectional.fromSTEB` (start/top/end/bottom) instead of `EdgeInsets.fromLTRB`.
- Use `AlignmentDirectional` instead of `Alignment` where direction matters.
- Don't use `Icons.arrow_forward` for "next" — use `Icons.arrow_forward_ios` and let Flutter mirror, or use `Directionality.of(context)` to pick the right icon.

We'll audit the merchant app for RTL-correctness before adding the first RTL locale.

---

## What we will not do

- **Translate emoji** (kept as-is across locales).
- **Translate brand names** (`MenuRay` is `MenuRay` everywhere).
- **Translate sample data** (`云间小厨` etc.) — these are intentionally Chinese to convey product context to Chinese-speaking developers; once we localize the demo / marketing site, we'll add per-locale demo data there.
- **Auto-translate without a paid LLM** (Google Translate API costs / accuracy / privacy concerns). The "Translate menu" button costs money per call; we'll surface that to merchants in their plan / quota.

## Common pitfalls

- **String concatenation breaks translation**:
  ```dart
  // ❌ Bad
  Text("Hello " + name + "!")
  
  // ✅ Good (in i18n once enabled)
  Text(t.hello(name: name))
  // arb: "hello": "Hello {name}!"
  ```
- **Hardcoded plural form**:
  ```dart
  // ❌ Bad
  Text("$count views")
  
  // ✅ Good — use plural placeholder
  ```
- **Mixed direction in RTL**: don't manually flip layouts; trust Flutter's directionality.
